# ADR-006: Investigation document format

**Status:** Accepted
**Date:** 2026-04-27
**Project:** war-room (Atlas Layer 3)

---

## Context

War-room investigations produce findings and a hypothesis iteratively (ADR-002).
When a PM requests a document or publishes an investigation, war-room must
synthesize the conversation into a structured, shareable artifact. ADR-001
characterized war-room as a "conversational + document" interface; this ADR
defines the document half: its format, its sections, and the skill responsible
for producing it.

---

## Decisions

### Format: Markdown

**Decision**: the investigation document is a Markdown string. The UI renders
it in the chat interface; the same string is stored verbatim in
`saved_investigations.document` (ADR-005).

Markdown rendering is a baseline capability of any web chat UI. Using it as
the document format means the investigation artifact is a single string that
serves both display and storage purposes with no intermediate conversion.

The document is human-readable in raw form. A PM who copies it out of war-room
receives a legible text artifact without rendering.

Rejected alternative — structured JSON with a frontend renderer: two artifacts
instead of one. Adds a rendering layer without MVP benefit. If richer
interactivity is needed (expandable sections, clickable release IDs), this is
the Phase 3 upgrade path.

Rejected alternative — HTML: not human-readable in raw form.

---

### Document sections

**Decision**: the investigation document follows this section order:

```
# <title>

**Question**: <original PM question, verbatim>

## Investigation
<one paragraph: which sources were queried, how many metrics investigated,
how many release candidates assessed, how many iterations used>

## Findings

### Metric findings
<funnel-investigation findings from context, one block per metric, preserving
the structured finding format from the funnel-investigation skill spec>

### Release candidates
<release-metric-correlation findings from context, preserving the structured
finding format. If release-metric-correlation was not invoked: "Not queried
in this investigation.">

## Hypothesis
<final hypothesis-formation output, verbatim — preserves the structured format:
Hypothesis, Confidence, Evidence for, Evidence against, What would confirm this,
What would refute this>

## Open questions
<bulleted list derived from the "Next steps" and "What would confirm/refute"
fields of the hypothesis>
```

The document records only the final hypothesis. The full investigation
evolution is in the conversation history (ADR-007) and accessible to the PM
while the conversation is active.

Rejected alternative — include all intermediate hypotheses: clutters the
document for the primary use case (sharing a clean summary with the team).
The conversation history serves the evolution use case for the investigating PM.

---

### Title generation

**Decision**: the title is auto-generated by `investigation-summary` as
`<metric_name> investigation — <YYYY-MM-DD>`, where `metric_name` is the
primary metric identified in the funnel-investigation findings. Falls back
to `Investigation — <YYYY-MM-DD>` if no metric name is identifiable.

The PM can override the title in the publish flow. If the PM confirms without
editing, the auto-generated title persists.

---

### Skill: `investigation-summary`

**Decision**: a new skill, `investigation-summary`, is responsible for
producing the investigation document. It is distinct from `hypothesis-formation`.

`hypothesis-formation` is a reasoning skill: invoked multiple times per
investigation, producing and refining a structured hypothesis for the
orchestrator's reasoning loop. Its output is an intermediate artifact consumed
by subsequent iterations.

`investigation-summary` is a synthesis-for-output skill: invoked once at the
end of an investigation, reads all findings and the final hypothesis from
context, and renders the complete human-facing document. It produces no
intermediate artifacts for further reasoning.

Keeping them separate preserves `hypothesis-formation`'s ability to be invoked
iteratively without generating a full document on every invocation. Each skill
has a single, clear output type.

This is the fifth skill in war-room's initial set (source-routing,
funnel-investigation, release-metric-correlation, hypothesis-formation,
investigation-summary), within the scope opened in ADR-001.

The skill specification is in `skills/investigation-summary/skill.md`,
produced as part of Phase 1b.

Rejected alternative — `hypothesis-formation` with a `document_mode`
parameter: coupling in-loop reasoning with end-of-loop rendering complicates
both modes and makes iterative invocation awkward.

---

### Trigger conditions

**Decision**: `investigation-summary` is invoked in two cases:

1. PM explicitly requests a document during or at the end of an investigation
   ("summarize this", "write this up", "give me a document").
2. Orchestrator triggers it automatically when the PM initiates a publish and
   `investigation-summary` has not yet been called in the current conversation.
   Every published investigation must have a document (ADR-005 schema requires it).

`investigation-summary` is not triggered automatically without an explicit
request or a publish action. Conversations remain document-free unless the PM
requests one or publishes.

If the PM publishes without a prior explicit document request, the orchestrator
auto-triggers `investigation-summary`, presents the document for review (with
title override option), then commits the publish.

---

## Consequences

**Positive**
- Single markdown string: no conversion between storage and display formats.
- Clean skill separation: `hypothesis-formation` stays a pure reasoning skill;
  `investigation-summary` is a pure synthesis skill.
- Auto-generation on publish ensures every published investigation has a
  document without a separate step from the PM.

**Negative / trade-offs**
- Only the final hypothesis is in the document. Investigation evolution is in
  the conversation history, not in the shared artifact.
- Title auto-generation degrades for vague questions ("why are numbers down?").
  PM override at publish time mitigates this.

**Constraints introduced**
- The publish flow must invoke `investigation-summary` if no document exists
  in the current conversation. The orchestrator cannot complete a publish
  without a document in context.
- `investigation-summary` depends on `hypothesis-formation` having been
  invoked at least once. If the PM attempts to publish before any hypothesis
  has been formed, the orchestrator must either prompt for hypothesis-formation
  first or explain that the investigation is not ready to publish.
- The document format (section order, heading levels) is the contract between
  `investigation-summary` and the UI renderer. Changes require updating both.
